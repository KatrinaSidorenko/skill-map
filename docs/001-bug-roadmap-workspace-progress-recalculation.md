# ADR-001: Roadmap Progress Recalculation Strategy

## Context & Problem
Currently, the progress of a roadmap (e.g., completed nodes vs. total nodes) is recalculated only when a new snapshot is created. Because snapshots are generated by a background process at specific intervals, the progress metrics are tightly coupled to the snapshot frequency. 

This causes a lag in the UI: when a user completes a node, the progress bar does not update immediately. We need a way to decouple progress calculation from the heavy snapshotting process to provide near real-time updates.

## Considered Options

### Option 1: Increase Snapshot Frequency
Trigger the background snapshot process more often (e.g., every minute or on every user action).

* **Pros:** * Very easy to implement. Reuses existing logic.
* **Cons:**
    * *Database Bloat:* Leads to massive database space consumption due to storing redundant full-state graphs.
    * *Performance Degradation:* Heavy CPU and I/O overhead. Snapshotting is an expensive operation; doing it constantly will slow down the primary database.
    * *Still not real-time:* If based on a cron job, there will always be a delay.

### Option 2: Dedicated Background Progress Service (Rehydration)
Extract a separate background service specifically for recalculating progress. It will take the last snapshot, apply all pending events (rehydrate the state), calculate the stats (total nodes, completed nodes), and save them to a metadata table.

* **Pros:** * Decouples snapshot logic from progress logic.
* **Cons:**
    * *Concurrency issues:* Potential race conditions if the Snapshot Taker and Progress Service try to read/write state simultaneously.
    * *Triggering dilemma:* When does it run? If it runs on a timer, progress is still delayed. If it runs on every event, it's doing too much heavy lifting (rebuilding the whole graph) just to increment a counter.
    * *Redundant computation:* Rehydrating the graph just to count nodes is highly inefficient.

### Option 3: Event-Driven Metadata Projection (Counters)
Create a separate metadata table (or add columns to the existing `Workspace` table). Initialize it when the first snapshot is made. Then, listen to domain events (`NODE_ADDED`, `NODE_DELETED`, `NODE_COMPLETED`) and directly increment or decrement the counters in the metadata table using a background worker or database triggers.

* **Pros:**
    * Extremely fast. No need to reconstruct the graph state.
    * Data is immediately available via a simple `JOIN` or direct query.
* **Cons & Trade-offs:**
    * *Concurrency (Your PostgreSQL concern):* What if `+` and `-` events arrive concurrently? 
        * *How Postgres handles it:* PostgreSQL is ACID-compliant. An operation like `UPDATE metadata SET total = total + 1 WHERE roadmap_id = X` acquires a row-level lock. Concurrent updates will wait in a queue and execute sequentially. It is safe from data corruption, but under *extreme* load (e.g., 1000 events/sec for one roadmap), this lock can cause a bottleneck.
    * *Event Ordering & Idempotency:* What happens if a `NODE_COMPLETED` event arrives *before* the `NODE_ADDED` event due to network lag? What if an event is processed twice? We might end up with negative counters or skewed statistics.
