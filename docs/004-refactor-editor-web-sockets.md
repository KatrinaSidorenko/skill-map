Introduce separate service for Web Sockets Service.
Possible name: LearningPlatform.Workspace.WebSockets.Service

For now, it can be implemented as a separate class library project in the solution. In the future it can be moved to a separate microservice if needed.

This service will be responsible for managing Web Socket connections related to the editor workspace. It will handle the communication events from the client and pass them to the queue for processing. Also should provide an interface for sending messages back to the client when necessary.

A used libraries:
- Microsoft.AspNetCore.SignalR in the backend
- @microsoft/signalr in the frontend

!JWT authentication:
.withUrl("/hub", { accessTokenFactory: () => myToken })

!Idempotency
- check the eventId generated by client


!Communication between WebSockets Service and Main Service:
- For now it can be implemented a main service can send messages to WebSockets via a in memory queue or a shared service. In the future, if WebSockets Service is moved to a separate microservice, communication can be done via HTTP API or message broker (e.g., RabbitMQ, Azure Service Bus).


The workflow:
1. Client establishes a Web Socket connection to the WebSockets Service when the editor workspace is opened.
2. Client sends events (e.g., code changes, cursor movements) to the WebSockets Service.
3. WebSockets Service receives the events and pushes them to a queue for processing.
4. Main Service processes the events from the queue and updates the workspace state accordingly.
5. Main Service sends messages about the updated state back to the WebSockets Service.
6. WebSockets Service forwards the messages to the client to update the editor workspace in real-time.

### Phase 1: Shared Contracts & Models
**Project Name Suggestion:** `Workspace.Core` or `Workspace.Shared`

* **`WorkspaceAction`**: Represents the *intent* from the user (e.g., "Add Edge"). Use "Action" or "Command" instead of "Event" here because it hasn't happened yet—it hasn't been validated.
* **`WorkspaceActionType`**: Enum (CreateNode, DeleteNode, CreateEdge, etc.).
* **`WorkspaceEvent`**: The *result* of a processed action. This is what gets saved to the DB and broadcasted back.

### Phase 5: Implementation of the "Hashed Lane"
Since you are starting in-memory, here is how to structure the "Traffic Lanes" for ordering:

```csharp
public class PartitionedActionStream : IWorkspaceActionStream
{
    private readonly Channel<WorkspaceAction>[] _lanes;
    private readonly int _laneCount = 4; // Start with 4 parallel lanes

    public PartitionedActionStream()
    {
        _lanes = Enumerable.Range(0, _laneCount)
            .Select(_ => Channel.CreateUnbounded<WorkspaceAction>())
            .ToArray();
    }

    public async Task PushAsync(WorkspaceAction action)
    {
        // Consistent Hashing: Same WorkspaceId always goes to the same lane
        var laneIndex = Math.Abs(action.WorkspaceId.GetHashCode()) % _laneCount;
        await _lanes[laneIndex].Writer.WriteAsync(action);
    }

    public ChannelReader<WorkspaceAction> GetReader(int laneIndex) => _lanes[laneIndex].Reader;
}
```

### Phase 6: Client-Side (Next.js)
1.  **`useWorkspaceSocket.ts`**: Hook to manage the `HubConnection`.
2.  **`ActionQueue`**: A local array of `pendingActions`.
3.  **`GraphReducer`**: Logic to update the UI optimistically.
    * **On Send:** Add to `pendingActions` + Update UI.
    * **On `OnActionConfirmed`:** Remove from `pendingActions` + Finalize UI state.
    * **On `OnActionRejected`:** Remove from `pendingActions` + Revert UI + Show Toast.
